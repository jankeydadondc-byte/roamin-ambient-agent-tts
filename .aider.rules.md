# Roamin Project Rules

## Terminal
You are running on Windows 11. Aider must be launched from PowerShell or CMD —
never from Git Bash. Git Bash's terminal emulation conflicts with Aider's UI.
If instructing the user to open a terminal, specify PowerShell.

## Git
Never auto-commit changes. Always show the diff and wait for the user to commit.
The project uses `no-auto-commits: true` in .aider.conf.yml — respect this intent.
Never amend a commit unless explicitly asked.
Never use destructive git commands (reset --hard, checkout --, clean -f) unless asked.
Never revert or discard changes you did not make. If the working tree has existing
changes in files you need to touch, read them and work with them — do not overwrite.
Always prefer non-interactive git commands.

## Behavior
- For simple, clear tasks: implement immediately — do not plan or ask permission.
  For ambiguous or large features: ask specific clarifying questions first, wait for
  answers before writing any code.
- No cheerleading, filler, or acknowledgement openers. Be direct and stay on task.
- Code review: only flag issues where confidence is high — you can cite exactly why
  it is wrong. False positives erode trust. Lead with bugs and risks ordered by
  severity with file:line references. Summaries follow findings, never precede them.
- Read a file before editing it. Group all edits to the same file into a single pass.
- If the same linter or type error recurs three times on the same file, stop and ask
  instead of looping.
- If unexpected changes in the working tree directly conflict with your current task,
  stop and ask. If they are in unrelated files, ignore them.

## Security
Flag and do not proceed if any of the following patterns appear in code being written:
- `eval()`, `new Function()` — code injection; find a safer alternative
- `os.system()` — shell injection; use `subprocess.run()` with a list of args
- `pickle` on untrusted data — arbitrary code execution; use JSON instead
- `.innerHTML =` or `dangerouslySetInnerHTML` with untrusted content — XSS;
  use `textContent` or DOMPurify
- `document.write()` — XSS and performance; use DOM methods instead
If genuinely required, flag it explicitly and let the user decide.

## Frontend
When working within the existing UIs (ui/control-panel or ui/roamin-chat), preserve
the established visual patterns, structure, and style — do not redesign what is there.

When creating new components or pages with no existing reference to match:
- Choose a clear aesthetic direction before writing a line of code and commit to it.
- Typography: choose distinctive fonts. Avoid Inter, Roboto, Arial, system-ui.
  Pair a display font with a refined body font.
- Color: use CSS variables for all values. Dominant colors with sharp accents
  outperform even palettes. Avoid purple gradients on white.
- Motion: one well-orchestrated page-load with staggered reveals beats scattered
  micro-interactions. Use the Motion library for React.
- Layout: prefer asymmetry, overlap, grid-breaking elements over centered boxes.
- Do not add useMemo/useCallback by default — only if already present in the file.
- Use modern React patterns (startTransition, useDeferredValue) where appropriate.
